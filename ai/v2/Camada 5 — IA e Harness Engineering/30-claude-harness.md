# Specforge V2 Claude Harness

## Status

Accepted

## Purpose

This document defines the Claude Harness used by Specforge.

The Claude Harness is responsible for transforming project knowledge, specifications and execution rules into a Claude-compatible operational environment.

Its purpose is to enable Claude Code to operate consistently within a project's engineering standards, architectural constraints and quality expectations.

The Claude Harness is an implementation of the generic Harness Model optimized for Claude.

---

# Introduction

Claude Code operates through context.

The quality of its outputs depends heavily on:

- available knowledge
- behavioral guidance
- operational constraints
- context quality

Most projects provide Claude with fragmented instructions.

Specforge introduces a structured Claude Harness that converts project knowledge into a durable operational environment.

The Claude Harness is not a prompt.

The Claude Harness is a project operating system for Claude.

---

# Definition

## Claude Harness

A Claude Harness is a Claude-compatible operational artifact generated from project knowledge and specifications.

A Claude Harness answers:

- What should Claude know?
- What should Claude prioritize?
- What behaviors are expected?
- What behaviors are forbidden?
- Which artifacts are authoritative?

The Claude Harness exists to align Claude behavior with project intent.

# Claude Harness Structure

## Project Identity

Defines:

- project purpose
- mission

- constraints

Purpose:

Prevent project drift.

---

## Project Context

Defines:

- active discoveries
- decisions
- context packages

Purpose:

Provide understanding.

---

## Authoritative Artifacts

Defines source-of-truth artifacts.

Examples:

- requirements
- architecture
- contracts
- security

Purpose:

Prevent conflicting interpretations.

---

## Behavioral Rules

Defines expected Claude behavior.

Examples:

- ask clarifying questions
- preserve traceability
- challenge assumptions
- reference specifications

Purpose:

Guide decision making.

---

## Execution Rules

Defines implementation expectations.

Examples:

- follow tasks
- preserve architecture
- generate tests
- preserve contracts

Purpose:

Guide execution.

---

## Validation Rules

Defines output quality expectations.

Examples:

- pass quality gates
- validate requirements
- validate contracts
- validate security requirements

Purpose:

Prevent low-quality output.

---

## Prohibited Behaviors

Examples:

- invent requirements
- bypass security
- ignore architecture
- remove traceability

Purpose:

Prevent harmful behavior.
