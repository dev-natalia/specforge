# Specforge V2 Product DNA

## Status

Accepted

## Purpose

This document defines the immutable identity of Specforge.

While features, architecture and implementation may evolve, the Product DNA should remain stable.

Its purpose is to ensure that future decisions remain aligned with the essence of the product.

Whenever uncertainty exists, Product DNA takes precedence over feature requests, implementation convenience and short-term opportunities.

---

# Product Identity

Specforge is a knowledge-first system for AI-assisted software development.

Its purpose is to help developers transform ideas into durable knowledge and durable knowledge into AI-ready context.

Specforge does not exist to generate code.

Specforge exists to improve the quality, consistency and reliability of AI-generated code by improving the quality of the context provided to AI systems.

---

# Core Belief

The quality of AI outputs is primarily determined by the quality of the context provided to the AI.

Most AI development problems are not model problems.

They are context problems.

Poor context creates poor outcomes.

Good context creates better outcomes.

Great context creates predictable outcomes.

Specforge exists to help developers produce great context.

---

# What We Are

Specforge is:

- A knowledge preservation system
- A specification generation system
- A context generation system
- A harness generation system
- A development governance system
- A bridge between human knowledge and AI implementation

Specforge helps developers think before they build.

Specforge helps AI systems understand before they generate.

---

# What We Are Not

Specforge is not:

- A code generation platform
- An IDE
- A project management system
- A ticketing system
- A documentation wiki
- A hosted SaaS dependency
- A replacement for software engineering knowledge
- A replacement for human decision making

Specforge should never become a generic productivity tool.

Specforge should remain focused on AI-assisted software development.

---

# Product Principles

## Principle 01 — Knowledge First

Knowledge is the primary asset.

Everything else is derived from knowledge.

Knowledge generates:

- Decisions
- Specifications
- Harnesses
- AI Context

Never optimize for specifications at the expense of knowledge.

## Principle 02 — Preserve Rationale

A decision without rationale is incomplete.

The system should preserve:

- Decisions
- Motivations
- Tradeoffs
- Alternatives
- Risks

Future developers must understand not only what was decided but why it was decided.

## Principle 03 — Durable Context

Context should survive beyond conversations.

Chat history is temporary.

Knowledge should be durable.

The value of Specforge increases as knowledge accumulates.

## Principle 04 — AI Agnostic

Specforge must remain independent of any AI provider.

Knowledge should outlive provider ecosystems.

## Principle 05 — Local First

Projects belong to users.

Knowledge belongs to users.

Generated artifacts belong to users.

## Principle 06 — Transparency

Users should understand:

- What was generated
- Why it was generated
- Which sources influenced generation

## Principle 07 — Simplicity Over Complexity

Additional complexity must provide measurable value.

---

# Anti-Patterns

## Prompt-Only Development

Relying entirely on prompts without preserving knowledge.

## Knowledge Compression

Reducing context so aggressively that rationale disappears.

## Tool Lock-In

Making projects dependent on a specific provider.

## Feature Accumulation

Adding features simply because they are possible.

## Spec-Centric Thinking

Treating specifications as the primary artifact.

Knowledge must always remain the source of truth.

---

# Decision Framework

Future features should be evaluated using the following questions.

1. Does this improve knowledge preservation?
2. Does this improve context quality?
3. Does this help developers produce better AI outcomes?
4. Does this introduce unnecessary complexity?
5. Does this align with Product DNA?
